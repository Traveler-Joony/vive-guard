# VibeGuard — 바이브 코딩 복잡도 가드레일 VS Code 확장

## 개발 기획서

---

## 1. 프로젝트 개요

### 제품명 (가칭)
**VibeGuard** — 바이브 코딩 프로젝트의 실시간 복잡도 모니터링 VS Code 확장

### 한 줄 설명
AI로 코드를 만들 때, 프로젝트가 "수정 스파이럴"에 빠지기 전에 경고해주는 체크엔진 라이트.

### 타겟 유저
바이브 코딩을 하는 개발자와 비전공자 (Cursor, Copilot, Claude 등 AI 코딩 도구 사용자)

### 지원 언어 (MVP)
JavaScript / TypeScript (React, Next.js, Vue 등 — 바이브 코딩에서 가장 많이 사용되는 스택)

## 2. 핵심 기능 정의

### 2.1 실시간 복잡도 대시보드 (사이드바 패널)

VS Code 사이드바에 전용 패널을 표시. Activity Bar에 아이콘 추가.

**표시 항목:**
- 전체 건강도 점수 (A~F 또는 0~100)
- 순환 복잡도 (Cyclomatic Complexity) 평균/최대
- 코드 중복률 (%)
- 패턴 일관성 점수 (%)
- 파일 간 의존도 (커플링 지수)
- 최근 변화 추세 (↑↓→ 화살표)

### 2.2 파일별 인라인 경고

에디터 상단 또는 CodeLens로 현재 파일의 복잡도를 표시.
- 함수별 복잡도 수치 (CodeLens)
- 임계값 초과 시 경고 아이콘

### 2.3 위험 감지 알림

특정 조건 충족 시 VS Code 알림(Notification)으로 경고:
- "dashboard.tsx의 복잡도가 25를 넘었습니다. 이 파일을 분리하는 것을 고려하세요."
- "에러 처리 패턴이 3가지로 혼재되어 있습니다."
- "프로젝트 복잡도가 급격히 증가 중입니다. 새 기능 추가 전에 리팩토링을 권장합니다."

### 2.4 액션 가이드 (비기술 언어)

경고와 함께 AI에게 복사해서 넣을 수 있는 리팩토링 프롬프트 제안:
- "AI에게 이렇게 요청해보세요: '모든 API 라우트의 에러 처리를 try-catch 패턴으로 통일해줘'"
- 클립보드 복사 버튼 포함

### 2.5 세이브 포인트 추천

복잡도가 급등하기 직전에 git commit을 권장하는 알림:
- "현재 상태를 저장하세요. 다음 변경이 위험할 수 있습니다."

### 3.1 전체 구조

```

├── src/
│   ├── extension.ts              # VS Code 확장 진입점 (activate/deactivate)
│   │
│   ├── analyzer/                  # 핵심 분석 엔진 (순수 로직, VS Code API 의존 없음)

│   │   ├── diagnosticsProvider.ts # Problems 패널 연동
```

### 3.2 핵심 설계 원칙

**1) 분석 엔진과 UI의 완전 분리**

`analyzer/` 폴더의 코드는 VS Code API에 일절 의존하지 않는 순수 TypeScript 모듈.
이유: 테스트가 쉽고, 나중에 웹앱이나 다른 에디터로 포팅 가능.

**2) 이벤트 기반 아키텍처**

```
파일 저장 이벤트
    → FileWatcher가 감지
    → Analyzer에 변경된 파일 전달
    → 해당 파일만 재분석 (전체 재분석 X)
    → HealthScorer가 종합 점수 업데이트
    → UI 컴포넌트들에 새 점수 전달
    → 사이드바/CodeLens/알림 갱신
```

**3) 증분 분석 (Incremental Analysis)**

파일 저장 시 변경된 파일만 재분석. 전체 프로젝트 재분석은 최초 1회 또는 수동 트리거.
이유: 대규모 프로젝트에서 성능 유지.

**4) 캐시 전략**

분석 결과를 메모리에 캐시. 파일 해시가 변경되지 않으면 재분석 스킵.

### 3.3 데이터 흐름

```
[VS Code Workspace]
        │
        ▼
[FileWatcher] ──파일 변경 감지──▶ [AST Parser]
                                      │
                                      ▼
                              ┌───────────────┐
                              │  분석 엔진     │
                              │               │
                              │ complexity    │──▶ 순환 복잡도
                              │ duplication   │──▶ 중복률
                              │ pattern       │──▶ 패턴 일관성
                              │ dependency    │──▶ 의존도 그래프
                              └───────┬───────┘
                                      │
                                      ▼
                              [HealthScorer]
                              종합 점수 계산
                                      │
                                      ▼
                    ┌─────────────────┼─────────────────┐
                    │                 │                 │
                    ▼                 ▼                 ▼
              [Sidebar]        [CodeLens]      [Notifications]
              대시보드          인라인 표시      경고 알림
```

## 4. 핵심 지표별 구현 전략

### 4.1 순환 복잡도 (Cyclomatic Complexity)

**구현 방법:** TypeScript Compiler API (`ts.createSourceFile`) 사용

```typescript
// 핵심 로직 개념
// AST를 순회하며 분기 노드를 카운트
const BRANCH_NODES = [
  ts.SyntaxKind.IfStatement,
  ts.SyntaxKind.ForStatement,
  ts.SyntaxKind.ForInStatement,
  ts.SyntaxKind.ForOfStatement,
  ts.SyntaxKind.WhileStatement,
  ts.SyntaxKind.DoStatement,
  ts.SyntaxKind.CatchClause,
  ts.SyntaxKind.ConditionalExpression, // 삼항 연산자
  ts.SyntaxKind.CaseClause,           // switch case
  ts.SyntaxKind.BinaryExpression,     // && || (논리 연산)
];
// 복잡도 = 1 + 분기 노드 수
```

**사용할 라이브러리:** `typescript` 빌트인 컴파일러 API (외부 의존성 최소화)

**임계값 기본값:**
- 🟢 1~10: 양호
- 🟡 11~20: 주의
- 🔴 21+: 위험

### 4.2 코드 중복률

**구현 방법:** 토큰 기반 슬라이딩 윈도우 해싱

```
1. 각 파일을 토큰 배열로 변환 (공백, 주석 제거)
2. N-토큰 윈도우(예: 30토큰)로 슬라이딩하며 해시 생성
3. 해시 충돌 = 중복 코드 블록
4. 중복 토큰 수 / 전체 토큰 수 = 중복률
```

**사용할 라이브러리:** 직접 구현 (가볍게) 또는 `jscpd` 코어 로직 참고

**임계값 기본값:**
- 🟢 0~5%: 양호
- 🟡 6~15%: 주의
- 🔴 16%+: 위험

### 4.3 패턴 일관성

**구현 방법:** 규칙 기반 패턴 매칭 (AST)

감지 대상:
- **에러 처리 패턴:** try-catch vs .catch() vs 미처리 → 비율 계산
- **상태 관리 방식:** useState vs Context vs 외부 라이브러리 혼용
- **API 호출 패턴:** fetch vs axios vs 자체 래퍼 혼용
- **export 스타일:** default export vs named export 혼용
- **파일 네이밍:** camelCase vs PascalCase vs kebab-case 혼용

```
일관성 점수 = (가장 많이 사용된 패턴의 비율)의 가중 평균
예: 에러 처리 10개 중 7개 try-catch, 2개 .catch, 1개 미처리
    → 에러 처리 일관성 = 70%
```

**임계값 기본값:**
- 🟢 80%+: 양호
- 🟡 60~79%: 주의
- 🔴 59% 이하: 위험

### 4.4 파일 간 의존도

**구현 방법:** import/require 구문 파싱 → 방향 그래프 구성

```
1. 모든 파일의 import/require 구문 수집
2. 방향 그래프(Directed Graph) 구성
3. 지표 계산:
   - 평균 in-degree (이 파일을 import하는 파일 수)
   - 순환 의존성 감지 (A→B→C→A)
   - "허브 파일" 감지 (in-degree가 비정상적으로 높은 파일)
   - 결합도 지수 = 전체 엣지 수 / (파일 수 × (파일 수-1))
```

**위험 신호:**
- 한 파일의 in-degree > 전체 파일 수의 30%
- 순환 의존성 존재
- 결합도 지수 > 0.3

## 5. 바이브 코딩 개발 시 예상 문제점 & 대책

이 프로젝트 자체를 바이브 코딩으로 만들 때, 다음 함정에 빠질 수 있다.

### 문제 1: VS Code Extension API의 특수성

**증상:** AI가 일반 Node.js 코드처럼 짜는데, VS Code 확장 환경에서 안 돌아감.

VS Code 확장은 일반적인 웹앱/Node.js와 다른 고유한 제약이 있다:
- Extension Host라는 별도 프로세스에서 실행됨
- Webview는 iframe 안에서 동작하며, 확장 본체와 postMessage로만 통신
- `vscode.workspace.fs`를 써야 하고 Node.js `fs`를 직접 쓰면 Remote/WSL 환경에서 실패
- activation event 설정이 잘못되면 확장이 아예 로드되지 않음

**대책:**
- `package.json`의 `activationEvents`, `contributes` 설정을 초반에 확정하고 AI에게 수정 금지 지시
- AI에게 프롬프트할 때 반드시 "이건 VS Code Extension이다. vscode API를 사용해라" 명시
- Webview ↔ Extension Host 통신 구조를 먼저 잡고 나서 기능 추가
- `package.json` 매니페스트를 "잠금 파일"로 취급 — AI가 함부로 건드리지 못하게 관리

### 문제 2: Webview 메시지 통신 꼬임

**증상:** 사이드바 UI가 데이터를 안 받거나, 이벤트가 이중 발생하거나, 메시지 타입이 뒤섞임.

Webview는 완전히 격리된 iframe이라 postMessage 외에 데이터 교환 방법이 없다. AI가 이 구조를 이해 못 하고 직접 함수 호출을 시도하거나, 메시지 타입 관리를 누락하는 경우가 많다.

**대책:**
- 메시지 프로토콜을 타입으로 먼저 정의 (변경 금지):
```typescript
// shared/messages.ts
type Message =
  | { type: 'UPDATE_HEALTH'; payload: HealthScore }
  | { type: 'UPDATE_FILE'; payload: FileAnalysis }
  | { type: 'REQUEST_REFRESH' }
  | { type: 'OPEN_FILE'; payload: { path: string } };
```
- 이 타입 파일을 AI에게 "이 파일은 절대 수정하지 마" 하고 프롬프트에 고정
- Webview 쪽 코드와 Extension Host 쪽 코드를 별도 프롬프트 세션으로 분리

### 문제 3: AST 파싱 성능

**증상:** 파일 저장할 때마다 전체 프로젝트를 분석하면서 에디터가 멈춤.

TypeScript AST 파싱은 파일당 10~100ms 걸릴 수 있고, 파일이 100개면 초 단위로 느려진다.

**대책:**
- 증분 분석 필수: 변경된 파일만 재분석, 결과 캐시
- 분석을 비동기로 실행 (`setTimeout` 또는 Web Worker 패턴)
- 디바운싱: 파일 저장 이벤트를 500ms 디바운스로 묶어서 연타 방지
- 초기 전체 분석은 프로그레스바와 함께 백그라운드 실행

### 문제 4: AI가 분석 엔진과 UI를 섞어버림

**증상:** `complexityAnalyzer.ts` 안에 `vscode.window.showInformationMessage`가 들어있고, UI 코드 안에 AST 파싱 로직이 섞여있음. 나중에 하나 수정하면 다른 쪽이 깨짐.

이것은 우리가 만드는 도구가 경고하려는 바로 그 문제 — 패턴 불일치와 높은 결합도.

**대책:**
- 디렉토리 구조를 먼저 만들고, 각 모듈의 인터페이스(입력/출력 타입)를 먼저 정의
- AI에게 파일별로 프롬프트: "complexityAnalyzer.ts만 구현해. vscode import 금지. 입력: 파일 경로 문자열, 출력: ComplexityResult 객체"
- 모듈 간 의존성 규칙을 CLAUDE.md나 프롬프트에 명시:
```
규칙:
- analyzer/ 폴더의 파일은 vscode를 import할 수 없다
- ui/ 폴더의 파일은 analyzer/를 직접 import할 수 없다 (healthScorer를 통해서만)
- watcher/ → analyzer → healthScorer → ui 순서의 단방향 의존성
```

### 문제 5: 테스트 코드 부재

**증상:** AI가 기능 코드만 짜고 테스트를 안 만듦. 리팩토링할 때 뭐가 깨졌는지 모름.

**대책:**
- 각 analyzer 모듈을 구현할 때 반드시 테스트 먼저 요청: "이 함수의 테스트를 먼저 짜줘. 그 다음에 구현해"
- `test/fixtures/`에 다양한 복잡도의 샘플 코드를 미리 준비
- VS Code 확장 테스트는 복잡하므로, analyzer 모듈은 순수 함수로 만들어 일반 단위 테스트로 검증

### 문제 6: package.json 매니페스트 오염

**증상:** AI가 기능 추가할 때마다 package.json의 contributes, activationEvents 등을 제멋대로 수정. 기존 등록된 명령이나 뷰가 사라짐.

VS Code 확장의 package.json은 일반 npm package.json보다 훨씬 복잡하다. `contributes` 아래에 commands, views, viewsContainers, configuration, menus 등이 엮여있고, 하나라도 잘못되면 확장이 로드 실패한다.

**대책:**
- package.json의 `contributes` 섹션을 초반에 완성하고, 이후 AI에게 "package.json은 내가 직접 수정한다. 네가 건드리지 마" 지시
- 변경이 필요하면 AI에게 "package.json에 추가해야 할 내용만 알려줘. 직접 수정하지 마" 요청

### 문제 7: 세션 간 컨텍스트 유실 (재준이 본인의 고통)

**증상:** 새 세션에서 AI가 프로젝트 구조를 모르고 엉뚱한 코드를 생성.

**대책:**
- 프로젝트 루트에 `ARCHITECTURE.md` 유지: 디렉토리 구조, 모듈 간 규칙, 현재 진행 상태
- 새 세션 시작 시 이 파일을 AI에게 먼저 읽히기
- Cursor 사용 시 `.cursorrules` 파일에 프로젝트 규칙 명시
- Claude Code 사용 시 `CLAUDE.md`에 동일 내용

## 7. 기술 스택

| 영역 | 기술 | 이유 |
|------|------|------|
| 언어 | TypeScript | VS Code 확장 공식 추천 + 타입 안전성 |
| AST 파싱 | TypeScript Compiler API (ts.createSourceFile) | 외부 의존성 없이 TS/JS 파싱 가능 |
| 복잡도 계산 | 직접 구현 | 라이브러리 의존성 최소화, 커스텀 가능 |
| 중복 감지 | 토큰 해싱 직접 구현 | jscpd는 무겁고, 핵심 알고리즘만 필요 |
| 사이드바 UI | Webview (HTML/CSS/JS) | VS Code 표준 방식 |
| 차트/시각화 | 직접 구현 (SVG/Canvas) 또는 Chart.js CDN | 가볍게 유지 |
| 테스트 | Mocha + VS Code Extension Test API | VS Code 확장 표준 테스트 프레임워크 |
| 빌드 | esbuild | webpack보다 빠름, 최신 VS Code 확장 추세 |
| 패키징 | vsce | VS Code 마켓플레이스 퍼블리시 표준 도구 |

## 8. 바이브 코딩 실행 전략

### AI에게 줄 프롬프트 순서

이 프로젝트를 바이브 코딩할 때 지켜야 할 프롬프트 순서:

**1단계 — 스캐폴딩 (AI와 함께)**
```
"VS Code 확장 프로젝트를 TypeScript로 초기화해줘. 
esbuild 번들러 사용. 다음 디렉토리 구조를 만들어줘: [구조 붙여넣기]
package.json에 다음 contributes를 설정해줘: [설정 붙여넣기]"
```

**2단계 — 타입 정의 (AI와 함께, 검증 필수)**
```
"다음 모듈들의 인터페이스 타입을 정의해줘. 구현은 하지 마. 타입만:
- ComplexityResult
- DuplicationResult  
- PatternConsistencyResult
- DependencyGraph
- HealthScore
- Message (Webview ↔ Extension 통신)"
```

**3단계 — 모듈별 구현 (하나씩, 테스트 포함)**
```
"complexityAnalyzer.ts를 구현해줘.
규칙:
- vscode를 import하면 안 됨
- 입력: 파일 경로 문자열 배열
- 출력: ComplexityResult[] (타입 파일 참고)
- TypeScript Compiler API 사용
- 테스트도 같이 만들어줘 (test/analyzer/complexity.test.ts)"
```

**4단계 — 통합 (조심스럽게)**
```
"fileWatcher.ts를 만들어줘. 
파일 저장 시 해당 파일만 analyzer에 전달해서 재분석.
500ms 디바운싱 적용.
package.json은 수정하지 마."
```

### 세션 관리 규칙

- 새 세션 시작 시 반드시 ARCHITECTURE.md를 먼저 공유
- 한 세션에서 2개 이상의 모듈을 동시에 작업하지 않기
- AI가 생성한 코드를 커밋하기 전에 다른 파일이 깨지지 않았는지 확인
- package.json 변경은 직접 수행하거나, AI의 제안을 검토 후 수동 적용

## 10. 리스크 & 완화

| 리스크 | 확률 | 영향 | 완화 |
|--------|------|------|------|
| AST 파싱이 모든 JS/TS 문법을 커버 못함 | 높음 | 중 | MVP에서는 React/Next.js 패턴에 집중. 나머지는 점진 추가 |
| Webview 통신이 복잡해져서 버그 다발 | 중 | 높 | 메시지 타입을 엄격하게 관리, 통신 로직을 단일 모듈로 격리 |
| 대규모 프로젝트에서 성능 이슈 | 중 | 높 | 증분 분석 + 캐싱 + 디바운싱으로 사전 대응 |
| AI가 프로젝트 구조를 깨뜨림 | 높음 | 높 | 모듈별 분리 개발, package.json 잠금, 인터페이스 먼저 정의 |
| 유사 도구 등장 | 낮음 | 중 | 빠르게 MVP 출시 + 커뮤니티 피드백 반영으로 선점 |
