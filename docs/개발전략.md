# VibeGuard — Claude Code 개발 전략

---

## 1. 핵심 원칙

Claude Code에서 가장 중요한 건 **컨텍스트 관리**다. 실패의 90%는 Claude가 프로젝트 맥락을 잃어버려서 생긴다.

세 가지를 확실하게 세팅하면 된다:
1. **CLAUDE.md** — 모든 세션에서 읽히는 프로젝트 뇌
2. **Skills** — 특정 작업 시에만 로드되는 전문 지식
3. **Hooks** — 실수를 자동으로 차단하는 가드레일

---

## 2. 프로젝트 초기 세팅 순서

### Step 1: 프로젝트 생성

```bash
# 빈 디렉토리에서 시작
mkdir vibeGuard && cd vibeGuard
git init

# Claude Code 시작
claude
```

첫 프롬프트:
```
VS Code 확장 프로젝트를 TypeScript로 초기화해줘.
- yo code 대신 수동으로 package.json, tsconfig.json 생성
- 번들러: esbuild
- 확장 이름: vibeGuard
- 아직 코드는 작성하지 마. 프로젝트 구조만 잡아줘.
```

### Step 2: CLAUDE.md 작성

프로젝트 루트에 직접 작성한다 (Claude에게 시키지 말고 직접).

```markdown
# VibeGuard — VS Code Extension for Vibe Coding Complexity Guardrails

## 프로젝트 개요
바이브 코딩 프로젝트의 복잡도를 실시간 모니터링하는 VS Code 확장.
JS/TS 코드를 정적 분석해서 순환 복잡도, 코드 중복, 패턴 일관성, 파일 의존도를 측정하고
사이드바 대시보드로 시각화한다.

## 기술 스택
- TypeScript (ES modules)
- VS Code Extension API
- TypeScript Compiler API (AST 파싱)
- esbuild (번들러)
- Mocha (테스트)

## 아키텍처 규칙 — IMPORTANT

### 의존성 방향 (단방향, 역방향 금지)
```
watcher/ → analyzer/ → ui/
```

### 절대 금지 사항
- analyzer/ 폴더의 파일은 `vscode`를 import할 수 없다
- ui/ 폴더의 파일은 analyzer/를 직접 import할 수 없다 (healthScorer.ts를 통해서만)
- `package.json`의 `contributes` 섹션을 수정하려면 반드시 나에게 먼저 알려줘
- Node.js `fs` 모듈 대신 반드시 `vscode.workspace.fs` 사용 (ui/ 내에서)
- Webview에서 Extension Host의 함수를 직접 호출 금지. postMessage만 사용

### 메시지 프로토콜
Webview ↔ Extension Host 통신은 `src/shared/messages.ts`의 타입만 사용.
이 파일은 내가 직접 관리한다. 수정 금지.

## 디렉토리 구조
```
src/
  extension.ts          — 진입점
  shared/
    types.ts            — 공용 타입 (분석 결과, 설정 등)
    messages.ts         — Webview 메시지 타입 (수정 금지)
  analyzer/             — 순수 TS, vscode 의존 없음
    complexityAnalyzer.ts
    duplicationDetector.ts
    patternChecker.ts
    dependencyMapper.ts
    healthScorer.ts
  watcher/
    fileWatcher.ts
  ui/
    sidebarProvider.ts
    codeLensProvider.ts
    diagnosticsProvider.ts
    notificationManager.ts
  config/
    thresholds.ts
  utils/
    astParser.ts
    tokenizer.ts
webview/                — 사이드바 HTML/CSS/JS
test/
  analyzer/             — 단위 테스트
  fixtures/             — 테스트용 샘플 코드
```

## 빌드 & 테스트 명령
```bash
npm run build          # esbuild 빌드
npm run watch          # 개발 모드
npm run test           # 전체 테스트
npm run test:unit      # analyzer 단위 테스트만
npm run lint           # eslint
npm run package        # .vsix 패키징
```

## 작업 규칙
- 새 모듈 구현 시 반드시 테스트부터 작성
- 한 번에 하나의 모듈만 작업
- 파일 생성/수정 후 반드시 `npm run build`로 빌드 확인
- 커밋 메시지: conventional commits (feat:, fix:, refactor:, test:)
```

### Step 3: 타입 정의 파일 직접 작성

CLAUDE.md에서 "수정 금지"로 명시한 파일들을 먼저 직접 만든다.

```
src/shared/types.ts와 src/shared/messages.ts를 만들어줘.

types.ts에는 다음 인터페이스를 정의:
- FileComplexity: 파일별 순환 복잡도 결과 (파일 경로, 함수별 복잡도 배열, 평균, 최대)
- DuplicationResult: 중복 코드 결과 (중복률 %, 중복 블록 목록)
- PatternConsistencyResult: 패턴 일관성 결과 (에러처리/export/네이밍 각각의 일관성 %)
- DependencyGraph: 의존성 그래프 (노드 목록, 엣지 목록, 순환 감지 결과)
- HealthScore: 종합 점수 (0~100, 등급 A~F, 4개 하위 점수, 경고 메시지 배열)
- AnalysisResult: 전체 분석 결과 (위 4개 + 타임스탬프)

messages.ts에는 Webview ↔ Extension 통신 메시지 타입:
- UPDATE_HEALTH: Extension → Webview (HealthScore 전달)
- UPDATE_FILES: Extension → Webview (파일별 상세 전달)  
- REQUEST_REFRESH: Webview → Extension (수동 재분석 요청)
- OPEN_FILE: Webview → Extension (파일 열기 요청)
- GET_SETTINGS: 양방향 (설정 조회/변경)
```

# VS Code Extension 개발 규칙

## Activation
- package.json의 activationEvents에 등록되지 않으면 확장이 로드되지 않음
- "onView:vibeGuard.dashboard"로 사이드바 열릴 때 활성화

## Webview
- Webview는 iframe으로 격리됨. Extension Host와 postMessage로만 통신
- Webview에서 vscode API 직접 접근 불가
- acquireVsCodeApi()는 Webview 내에서 한 번만 호출

## CodeLens
- CodeLensProvider를 등록하고 package.json contributes.languages에 설정
- 문서 변경 시 onDidChangeCodeLenses 이벤트 발생시켜야 갱신됨

## FileSystemWatcher
- vscode.workspace.createFileSystemWatcher 사용
- glob 패턴: `**/*.{ts,tsx,js,jsx}`
- onDidChange, onDidCreate, onDidDelete 이벤트 핸들링

## 성능
- Extension Host는 싱글 스레드. 무거운 작업은 비동기로
- 디바운싱 필수: 파일 저장 이벤트를 500ms 묶기
- 증분 분석: 변경된 파일만 재분석, 결과 캐시
```

# Analyzer 모듈 규칙

## 핵심 원칙
- vscode import 절대 금지. 순수 TypeScript만 사용
- 입력: 파일 경로 문자열 또는 소스 코드 문자열
- 출력: src/shared/types.ts에 정의된 타입 객체

## AST 파싱
- TypeScript Compiler API 사용: ts.createSourceFile()
- ScriptTarget.Latest, jsx: JsxEmit.React 설정
- sourceFile.forEachChild()로 재귀 순회

## 순환 복잡도 계산
분기 노드: IfStatement, ForStatement, ForInStatement, ForOfStatement,
WhileStatement, DoStatement, CatchClause, ConditionalExpression,
CaseClause, BinaryExpression(&&, ||)
복잡도 = 1 + 분기 노드 수 (함수 단위)

## 중복 감지
토큰화 → 공백/주석 제거 → N-토큰(30) 슬라이딩 윈도우 → 해시 비교
해시 충돌 = 중복 블록. 중복률 = 중복 토큰 / 전체 토큰

## 패턴 일관성
감지 대상:
- 에러 처리: try-catch vs .catch() vs 미처리
- export 스타일: default vs named
- 파일 네이밍: camelCase vs PascalCase vs kebab-case
가장 많은 패턴의 비율 = 일관성 점수

## 의존성 맵
import/require 구문 파싱 → 방향 그래프 → in-degree, 순환 감지
```

### Step 5: Hooks 설정

Claude가 실수하기 쉬운 패턴을 자동 차단한다.

프롬프트로 설정:
```
다음 hook들을 설정해줘:

1. PostToolUse hook (파일 수정 후):
   - src/analyzer/ 디렉토리의 파일에 'from "vscode"' 또는 "from 'vscode'"가
     포함되면 에러 메시지 출력: "analyzer 모듈에서 vscode import 금지"
   
2. PostToolUse hook (파일 수정 후):
   - src/shared/messages.ts가 수정되면 경고:
     "messages.ts는 수동 관리 파일입니다. 변경이 의도된 것인지 확인하세요."

3. PostToolUse hook (파일 수정 후):
   - package.json이 수정되면 경고:
     "package.json이 변경되었습니다. contributes 섹션을 확인하세요."
```

## 4. 세션 관리 규칙

### 언제 /clear 하는가
- 새 모듈 작업 시작할 때
- 컨텍스트 사용량이 50% 넘었을 때 (수동 `/compact`도 가능)
- 디버깅이 3회 이상 실패해서 맴돌 때

### 언제 /compact 하는가
- 같은 모듈 안에서 대화가 길어졌지만 아직 작업 중일 때
- 테스트 실패 → 수정 → 재실행 루프가 반복될 때

### 커밋 전략
- 각 모듈 완성 + 테스트 통과 시마다 커밋
- Claude Code에게 커밋 시키기:
```
지금까지 변경사항을 커밋해줘.
메시지: "feat: implement complexity analyzer with unit tests"
```

### 디버깅 루프 탈출 규칙
같은 에러를 3번 이상 고치려고 시도했으면:
1. `/clear`로 컨텍스트 초기화
2. 에러 메시지만 정확히 복사해서 새 세션에서 질문
3. 그래도 안 되면 해당 모듈만 별도로 빼서 최소 재현 케이스 만들기

### CLAUDE.md 관리

- "AI가 틀리는 것"만 적기. 당연한 건 적지 않기

### 세션당 하나의 작업
- 한 세션에서 analyzer와 UI를 동시에 만들면 안 됨
- 컨텍스트가 섞이면서 analyzer에 vscode import가 들어가는 사고 발생
- 모듈 간 경계를 세션 경계와 일치시키기

### Plan → Execute 패턴
- 모든 작업에서 plan 모드를 먼저 요청
- 계획을 검토한 후에만 실행 허가
- 복잡한 작업일수록 이 패턴의 가치가 큼

### 점진적 통합
- 각 모듈이 독립적으로 테스트를 통과한 후에만 통합
- 통합 시에도 한 번에 하나씩 연결
- 통합할 때마다 전체 테스트 실행
